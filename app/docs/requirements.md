## 1. 项目概述

### 1.1 项目背景

DeepAgents 是一个基于 FastAPI 构建的 AI Agent 服务平台，旨在为 AI 应用提供标准化的后端服务能力。项目支持多沙箱后端（OpenSandbox / Daytona / Modal），通过 RESTful API 和 SSE 流式接口对外暴露 AI Agent 能力，同时提供完整的用户管理和任务队列功能。

### 1.2 项目目标

- 提供标准化的用户管理 API（注册、认证、CRUD）
- 提供可靠的任务队列系统，支持状态机管理、重试、取消
- 集成多种沙箱后端，通过统一接口调用 AI Agent
- 支持 SSE 流式输出，适配实时对话场景
- 为前端和第三方客户端提供完整的 OpenAPI 文档

### 1.3 核心价值

| 维度 | 说明 |
|------|------|
| **多后端支持** | 统一的 Agent 接口，底层支持 OpenSandbox / Daytona / Modal |
| **异步优先** | 全链路异步设计，数据库采用 async SQLAlchemy，最大化并发吞吐 |
| **状态机驱动** | 任务队列基于状态机设计（pending → running → success/failed/cancelled） |
| **开发友好** | 自动生成 OpenAPI 文档（Swagger UI + ReDoc），支持热重载启动 |

## 3. 用户故事

### 3.1 用户注册与认证

**角色**: 开发者 / API 消费者  
**场景**: 第三方客户端需要调用本服务，必须先创建用户账号

```
作为 API 消费者
我想要注册一个新账号
以便我可以创建任务和使用 Agent 服务

验收标准:
- 用户名唯一，不允许重复注册
- 邮箱格式正确且唯一
- 密码至少 6 位
- 注册成功后返回用户信息（不含密码）
```

### 3.2 任务创建与执行

**角色**: 任务执行器（Worker）  
**场景**: 后端任务执行器从队列中拉取任务，执行完成后更新状态

```
作为任务执行器
我想要查询 pending 状态的任务
以便我可以获取待执行的工作单元

作为任务执行器
我想要更新任务状态为 running / success / failed
以便我可以报告任务执行结果

验收标准:
- 状态流转严格遵循状态机规则
- 状态变更自动记录 started_at / completed_at 时间戳
- 执行失败时支持填入 error 信息
- 支持通过 progress 字段更新进度 0-100
```

### 3.3 Agent 对话（普通 + 流式）

**角色**: 前端应用 / 第三方客户端  
**场景**: 客户端通过 API 与 AI Agent 进行实时对话

```
作为前端应用
我想要发送消息给 Agent 并获取回复
以便我可以实现聊天功能

作为前端应用
我想要以 SSE 流式方式接收 Agent 回复
以便我可以实现打字机效果的实时显示

验收标准:
- 普通对话返回完整消息内容
- 流式对话通过 SSE 逐段推送内容片段
- 流式结束时发送 `data: [DONE]\n\n`
- 支持指定沙箱后端（opensandbox / daytona / modal）
```

### 3.4 任务取消与重试

**角色**: 用户  
**场景**: 用户发现自己提交的任务有问题，想要取消或重试

```
作为用户
我想要取消一个还在排队（pending）或执行中（running）的任务
以便我可以停止不需要的任务

作为用户
我想要重试一个失败（failed）的任务
以便我可以尝试修复问题后重新执行

验收标准:
- 只有 pending/running 状态可以取消
- 只有 failed 状态可以重试
- 重试次数不得超过 max_retries
- 每次重试 retry_count +1
```

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| API 响应时间 | P95 < 500ms | 不含 Agent 对话（沙箱调用耗时不定） |
| 数据库查询 | P95 < 100ms | SQLite 单文件，async 模式 |
| 并发支持 | 100+ 并发连接 | 依赖 uvicorn 异步架构 |
| SSE 连接保持 | 支持长连接 | `X-Accel-Buffering: no` 禁用 nginx 缓冲 |

### 4.2 安全需求

| 方面 | 当前实现 | 建议改进 |
|------|----------|----------|
| 密码存储 | SHA256 简单哈希 | 生产环境替换为 bcrypt 或 argon2 |
| CORS | `allow_origins=["*"]` | 生产环境限制为具体域名 |
| 认证 | 无（预留扩展点） | 建议接入 JWT / OAuth2 |
| 输入校验 | Pydantic 模型自动校验 | 已实现，需保持覆盖 |
| SQL 注入 | ORM 参数化查询 | 已实现 |

### 4.3 兼容性需求

| 项目 | 要求 |
|------|------|
| Python 版本 | >= 3.10 |
| 数据库 | SQLite（开发）、PostgreSQL/MySQL（生产可通过修改 DATABASE_URL） |
| 浏览器兼容 | 现代浏览器（Chrome/Firefox/Safari/Edge） |
| API 兼容性 | REST + SSE，行业标准 |

### 4.4 可维护性需求

| 方面 | 实现方式 |
|------|----------|
| 代码结构 | 分层清晰（routers / models / schemas / core / sandbox） |
| 文档 | 自动生成 Swagger UI + ReDoc |
| 日志 | 可通过 uvicorn 日志配置 |
| 配置管理 | `pydantic-settings` + `.env` 文件 |

## 5. 约束条件

### 5.1 技术约束

- **框架限制**: 选用 FastAPI，因此必须基于 Python 生态
- **数据库限制**: 预设为 SQLite，适合开发和轻量部署；生产需切换到 PostgreSQL
- **沙箱后端**: OpenSandbox API Key 必须配置才能使用 Agent 功能；Daytona/Modal 为预留后端，SDK 尚未接入
- **异步约束**: 所有数据库操作使用 async SQLAlchemy；同步库（如 `requests`）不可用

### 5.2 时间约束

- 本文档为 2026-03-28 时的快照，随着项目迭代，部分功能（如 Daytona/Modal 支持）将逐步实现

### 5.3 资源约束

- SQLite 单文件数据库，无高可用设计，不适合大规模多节点部署
- Agent 沙箱冷启动有延迟（尤其是 Daytona 和 Modal），设计时应考虑预热机制
